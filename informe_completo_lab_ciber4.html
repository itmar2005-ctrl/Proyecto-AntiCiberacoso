Sección 01

Resumen Ejecutivo

Durante el laboratorio se trabajó en dos frentes paralelos. El primero fue la defensa del WordPress propio, expuesto por Tailscale, aplicando controles de Apache, PHP, WordPress, ModSecurity, fail2ban, iptables, mod_evasive, honeypots, 2FA y monitoreo. El segundo fue la auditoría ofensiva autorizada contra el WordPress objetivo http://100.72.233.79:8080/, donde se identificaron vulnerabilidades críticas y altas relacionadas con CORS, TLS, REST API, headers de proxy, Host Header Injection, plugins y rutas administrativas.

En defensa, se corrigieron dos fallos críticos: un rate-limit de ModSecurity inoperante por fases mal ubicadas y un bypass de fail2ban causado por la cadena ts-input de Tailscale. Después de las correcciones, las tres capas principales quedaron operativas: ModSecurity a nivel aplicación, fail2ban a nivel red y mod_evasive a nivel Apache.

En ofensiva, se confirmaron 16 hallazgos principales contra el objetivo. Los más graves fueron CORS misconfiguration con credenciales, ausencia de TLS, confianza en X-Forwarded-Proto, endpoint /batch/v1 con POST habilitado, Host Header Injection con Open Redirect, usuarios expuestos vía REST API y múltiples plugins/namespaces detectados.

Riesgo global del objetivo auditado: Crítico

El riesgo global del WordPress auditado se considera crítico por la sinergia entre vulnerabilidades: CORS permite leer respuestas cross-origin con credenciales; la falta de TLS expone credenciales en tránsito; REST API revela usuarios y endpoints; /batch/v1 permite operaciones en lote; Host Header Injection y X-Forwarded-Proto permiten manipular URLs y redirecciones. Cada vulnerabilidad individual es seria, pero encadenadas elevan considerablemente el impacto.

Sección 05

Bugs Críticos Corregidos

Bug 1 · Rate-limit ModSecurity inoperante

El rate-limit de login estaba definido con incremento de contador en una fase posterior al chequeo. Esto hacía que la regla de bloqueo no actuara correctamente durante la evaluación de la petición. La corrección consistió en mover la inicialización, el incremento y el chequeo a una fase temprana, permitiendo que el contador estuviera disponible antes de decidir si bloquear.

AntesDespuésResultado
Contador incrementado tardeContador incrementado y evaluado tempranoTras varios intentos, ModSecurity responde 429

Bug 2 · fail2ban era bypassed por Tailscale

La cadena ts-input de Tailscale aceptaba todo el tráfico de tailscale0 antes de que fail2ban pudiera evaluar reglas de bloqueo. Esto dejaba sin efecto los jails contra tráfico proveniente de la tailnet. La solución fue insertar las cadenas de fail2ban dentro de ts-input antes del ACCEPT general.

ProblemaCorrecciónResultado
ACCEPT general de Tailscale antes de fail2banfail2ban se evalúa antes del ACCEPTIPs baneadas son rechazadas correctamente

V-01 · CORS misconfiguration

La REST API refleja cualquier Origin y permite Access-Control-Allow-Credentials: true. La prueba se realizó enviando un origen controlado y observando que el servidor lo devolvía en los headers CORS junto con credenciales permitidas.

Access-Control-Allow-Origin: http://evil.com
    
V-02 · Ausencia total de TLS/HTTPS
El sitio funciona por HTTP en el puerto 8080. Esto implica que credenciales, cookies, formularios y respuestas REST no tienen cifrado de aplicación.
Impacto: exposición de credenciales y sesiones dentro de la red si existe observación de tráfico.
Mitigación: habilitar HTTPS, redirigir HTTP, configurar HSTS y cookies Secure.

    
NF-02 · X-Forwarded-Proto confiado
Al enviar X-Forwarded-Proto: https, WordPress cambia la generación de URLs absolutas de HTTP a HTTPS. Esto indica que el servidor confía en un header manipulable por el cliente.
Impacto: combinado con Host Header Injection permite poisoning de URLs y redirecciones.
Mitigación: confiar en headers proxy solo desde proxies conocidos; fijar WP_HOME y WP_SITEURL.

    
NF-03 · /batch/v1 acepta POST
El endpoint REST batch permite POST. Este endpoint puede agrupar múltiples operaciones REST. El riesgo aumenta al combinarse con CORS mal configurado.
Impacto: potencial CSRF batch contra REST API si existe una sesión autenticada.
Mitigación: restringir el endpoint, exigir nonces, limitar métodos y corregir CORS.

    
NF-04 · wordfence/v1/authenticate acepta POST
El endpoint de autenticación de Wordfence expone método POST según OPTIONS. No se ejecutó explotación destructiva, pero se documentó la superficie.
Impacto: superficie de autenticación adicional, combinable con usuario enumerado, CORS y falta de TLS.
Mitigación: actualizar Wordfence, verificar controles de autenticación y rate limiting.

    
V-03 · Wordfence namespace expuesto
El namespace wordfence/v1 es visible sin autenticación, incluyendo rutas de autenticación, configuración y escaneo.
Impacto: revela plugin de seguridad, rutas internas y superficie para análisis de CVEs.
Mitigación: restringir namespaces REST no requeridos y mantener plugin actualizado.

    
V-04 · install.php, upgrade.php y repair.php accesibles
Rutas administrativas de WordPress accesibles públicamente. Aunque no siempre permiten cambios, revelan estado del sistema y superficie administrativa.
Mitigación: bloquear o restringir estas rutas con autenticación/ACL.

    
V-05 · functions.php retorna HTTP 500
El acceso directo al archivo del tema provoca error 500. Esto sugiere ejecución directa no controlada o dependencia de contexto WordPress.
Impacto: potencial Full Path Disclosure si errores PHP se muestran en otra configuración.
Mitigación: proteger archivos PHP del tema, desactivar display_errors e investigar logs internos.

    
V-06 · Host Header Injection + Open Redirect
El servidor usa el header Host para construir redirecciones. Con un Host controlado, las redirecciones pueden apuntar a dominios externos.
Impacto: phishing, poisoning de enlaces y abuso de flujos de login.
Mitigación: validar Host con allowlist y fijar URL canónica en WordPress/Apache.

    
V-07 · REST API expone usuario c4_zheta_mny
El endpoint /wp/v2/users expone el usuario sin autenticación.
Impacto: facilita ataques dirigidos, password spraying o phishing.
Mitigación: restringir REST users y bloquear author enumeration.

    
NF-01 · Plugins nuevos descubiertos
Se detectaron Burst Statistics, All In One Security, Advanced IP Blocker y WP-Optimize, además de Wordfence y Akismet probable.
Impacto: cada plugin suma endpoints, archivos públicos y posibles CVEs.
Mitigación: inventario, actualización, eliminación de plugins innecesarios y revisión de endpoints.

    
V-08 · Enumeración de usuarios por ?author=N
El patrón ?author=N permite distinguir IDs existentes por redirecciones/respuestas.
Mitigación: bloquear author enumeration en servidor o WordPress.

    
V-09 · wp-cron.php accesible
wp-cron.php responde públicamente. Puede ser abusado para generar carga si se invoca repetidamente.
Mitigación: deshabilitar WP-Cron público y usar cron del sistema.

    
V-10 · Cookies sin SameSite y falta CSP
Cookies sin SameSite y ausencia de CSP en home incrementan riesgo de CSRF, clickjacking o explotación complementaria.
Mitigación: SameSite=Lax/Strict, Secure, HttpOnly y CSP completa.

    
V-11 · URLs internas localhost:8080
El HTML, headers y REST revelan localhost:8080. Esto muestra mala configuración de siteurl/home.
Mitigación: corregir WP_HOME, WP_SITEURL y reemplazar URLs internas en base de datos.

Sección 12

Fichas Detalladas de Ataques, Pruebas y Resultados

Esta sección amplía cada prueba importante con el mismo nivel de detalle del informe Red Team original. Cada ficha explica qué se intentó, por qué se probó, cómo se verificó de forma controlada, qué se encontró, qué impacto tiene y cómo debe mitigarse.

Bloque A · Pruebas ofensivas contra el WordPress objetivo

A-01 · Prueba CORS con Origin malicioso

Objetivo: verificar si la REST API aceptaba peticiones cross-origin desde dominios no autorizados.

Procedimiento: se envió una petición HTTP a la REST API usando un header Origin controlado, simulando que la petición venía desde un sitio externo como evil.com. Luego se revisaron los headers de respuesta.

curl -s -D - -o /dev/null -H "Origin: http://evil.com" \
  'http://100.72.233.79:8080/index.php?rest_route=/'

Resultado: el servidor reflejó el origen enviado y además permitió credenciales.

Access-Control-Allow-Origin: http://evil.com
Access-Control-Allow-Credentials: true
Access-Control-Allow-Methods: OPTIONS, GET, POST, PUT, PATCH, DELETE

Qué reveló: la política CORS no valida una lista blanca de orígenes y permite credenciales cross-origin.

Efectividad: efectiva. Confirmó vulnerabilidad crítica.

Impacto: si un usuario autenticado visita un sitio malicioso, ese sitio podría hacer peticiones a la REST API con credenciales del usuario.

Mitigación: permitir solo orígenes explícitos, no reflejar Origin dinámicamente y no usar Access-Control-Allow-Credentials: true salvo necesidad justificada.

A-02 · Verificación de ausencia de TLS/HTTPS

Objetivo: confirmar si el WordPress protegía credenciales y cookies mediante HTTPS.

Procedimiento: se accedió al sitio por su URL principal y se verificó que el esquema era http:// sobre el puerto 8080. También se revisaron cookies emitidas por WordPress.

curl -i 'http://100.72.233.79:8080/wp-login.php' | grep -i 'set-cookie'

Resultado: el sitio opera por HTTP y las cookies observadas no tienen protección Secure.

Qué reveló: las credenciales de login y cookies pueden viajar sin cifrado de aplicación.

Efectividad: efectiva. Confirmó debilidad crítica de transporte.

Impacto: exposición de credenciales y sesión ante observación de tráfico en el entorno de red.

Mitigación: habilitar HTTPS, redirigir HTTP a HTTPS, activar HSTS y configurar cookies con Secure.

A-03 · Host Header Injection y Open Redirect

Objetivo: determinar si WordPress/Apache construía redirecciones usando el header Host enviado por el cliente.

Procedimiento: se envió una petición a /wp-admin/ usando Host: evil.com y se analizó el header Location.

curl -s -D - -o /dev/null -H "Host: evil.com" \
  'http://100.72.233.79:8080/wp-admin/' | grep -i '^Location:'

Resultado: el servidor generó una redirección hacia el dominio indicado en el Host.

Location: http://evil.com/wp-admin/?redirect_to=http://evil.com/wp-admin/&reauth=1

Qué reveló: el sitio no valida el Host contra una lista blanca y lo usa para generar URLs absolutas.

Efectividad: efectiva. Confirmó Open Redirect y Host Header Injection.

Impacto: phishing, poisoning de enlaces, abuso de redirecciones de login y posible cache poisoning si existiera proxy intermedio.

Mitigación: configurar ServerName/ServerAlias, validar Host en Apache/WordPress y fijar WP_HOME/WP_SITEURL.

A-04 · Manipulación de X-Forwarded-Proto

Objetivo: verificar si WordPress confiaba en headers de proxy controlados por el cliente.

Procedimiento: se comparó la página normal contra la misma página con X-Forwarded-Proto: https.

curl -s -H "X-Forwarded-Proto: https" \
  'http://100.72.233.79:8080/' | grep -o 'https://[^" ]*' | head

Resultado: las URLs absolutas del sitio cambiaron de http:// a https://.

Qué reveló: el servidor confía en X-Forwarded-Proto sin validar si la petición viene de un proxy legítimo.

Efectividad: efectiva. Confirmó header proxy trust inseguro.

Impacto: combinable con Host Header Injection para poisoning de URLs absolutas y redirecciones.

Mitigación: aceptar headers proxy solo desde IPs de proxies confiables y definir URLs canónicas fijas.

A-05 · Enumeración de usuarios vía REST API

Objetivo: comprobar si WordPress exponía usuarios sin autenticación mediante REST API.

Procedimiento: se consultó /wp/v2/users usando rest_route.

curl -s 'http://100.72.233.79:8080/index.php?rest_route=/wp/v2/users/'

Resultado: se obtuvo información del usuario c4_zheta_mny.

id: 1
name: c4_zheta_mny
slug: c4_zheta_mny

Qué reveló: nombre y slug de usuario válido sin autenticación.

Efectividad: efectiva. Confirmó exposición de usuario.

Impacto: facilita ataques dirigidos contra login, phishing o password spraying.

Mitigación: restringir REST users para usuarios no autenticados y bloquear enumeración de autores.

A-06 · Enumeración por author redirect

Objetivo: verificar si el patrón ?author=N permite descubrir IDs de usuarios.

Procedimiento: se consultaron IDs secuenciales y se compararon códigos, redirects y tamaños.

for i in 1 2 3 5 10 20 30; do
  curl -s -o /dev/null -w "author=$i %{http_code}\n" \
    "http://100.72.233.79:8080/?author=$i"
done

Resultado: 29+ IDs presentaron comportamiento de usuario existente.

Qué reveló: permite mapear cuentas internas aunque REST users se corrigiera después.

Efectividad: efectiva.

Mitigación: bloquear ?author= y usar respuestas uniformes para IDs inexistentes.

A-07 · Enumeración de namespaces REST y plugins

Objetivo: descubrir plugins y funcionalidades expuestas mediante REST API.

Procedimiento: se descargó el índice REST y se extrajeron namespaces/rutas.

curl -s 'http://100.72.233.79:8080/index.php?rest_route=/' | \
  python3 -c "import sys,json; d=json.load(sys.stdin); print('\n'.join(d.get('routes',{}).keys()))"

Resultado: se identificaron namespaces de Wordfence, Burst Statistics, AIOS, Advanced IP Blocker y WP-Optimize.

Qué reveló: plugins instalados y endpoints adicionales, algunos con métodos POST.

Efectividad: muy efectiva. Permitió descubrir hallazgos NF-01, NF-03 y NF-04.

Mitigación: restringir REST API de plugins, eliminar plugins no usados y mantenerlos actualizados.

A-08 · Permission matrix con OPTIONS

Objetivo: detectar endpoints REST con métodos sensibles permitidos sin ejecutar cambios.

Procedimiento: se usó OPTIONS contra rutas REST para observar métodos permitidos, sin enviar mutaciones reales.

curl -s -X OPTIONS -D - -o /dev/null \
  'http://100.72.233.79:8080/index.php?rest_route=/batch/v1'

Resultado: /batch/v1 acepta POST y /wordfence/v1/authenticate acepta GET/POST.

Qué reveló: endpoints capaces de recibir operaciones activas.

Efectividad: efectiva y segura, porque no modificó datos.

Mitigación: limitar métodos, exigir autenticación, nonces y corregir CORS.

A-09 · Rutas administrativas expuestas

Objetivo: verificar si archivos administrativos de WordPress estaban accesibles desde internet/tailnet.

Procedimiento: se consultaron rutas de instalación, actualización y reparación.

for f in install.php upgrade.php repair.php setup-config.php; do
  curl -s -o /dev/null -w "%{http_code} $f\n" \
    "http://100.72.233.79:8080/wp-admin/$f"
done

Resultado: install.php, upgrade.php y repair.php respondieron públicamente.

Qué reveló: exposición de superficie administrativa innecesaria.

Efectividad: efectiva.

Mitigación: bloquear estos archivos con reglas de Apache o autenticación.

A-10 · Análisis de cookies y CSP

Objetivo: revisar flags de cookies y headers de seguridad.

Procedimiento: se observaron headers de /wp-login.php y home.

curl -sI 'http://100.72.233.79:8080/wp-login.php' | grep -i 'set-cookie'
curl -sI 'http://100.72.233.79:8080/' | grep -i 'content-security'

Resultado: cookies sin SameSite/Secure y ausencia de CSP en home.

Qué reveló: hardening incompleto de headers y cookies.

Efectividad: efectiva.

Mitigación: cookies con SameSite/Secure y CSP en todas las páginas.

A-11 · Detección de localhost:8080 en URLs

Objetivo: comprobar si WordPress estaba configurado con URL interna.

Procedimiento: se extrajeron URLs absolutas del HTML y REST API.

curl -s 'http://100.72.233.79:8080/' | \
  grep -o 'http://localhost:8080[^" ]*' | sort -u | head

Resultado: se encontraron múltiples enlaces hacia localhost:8080.

Qué reveló: mala configuración de siteurl/home.

Mitigación: corregir URL base en WordPress y reemplazar referencias antiguas.

Bloque C · Ataques recibidos contra el WordPress propio

C-01 · ffuf fuzzing contra archivos y backups

Atacante observado: 100.115.41.66.

Qué hizo: envió miles de requests buscando rutas y archivos como .sql, .rar, .zip, .bak.

Qué detectó la defensa: User-Agent/patrón de ffuf y extensiones sensibles.

C-02 · Hydra brute force contra wp-login

Atacante observado: 100.107.30.34.

Qué detectó la defensa: intentos fallidos repetidos y User-Agent ofensivo.

Respuesta: fail2ban registró fallos y aplicó ban; ModSecurity también tenía rate-limit.

Lección: rate-limit de aplicación y ban de red se complementan.

C-03 · WPScan/Nmap probing

Atacante observado: 100.110.242.122.

Qué hizo: probing de plugins, themes y scripts de reconocimiento.

Respuesta: ModSecurity bloqueó User-Agents ofensivos y acceso a rutas de themes/plugins no permitidas.

Lección: los User-Agents son fáciles de cambiar, pero aun así sirven como señal temprana combinados con comportamiento.

C-04 · Pruebas de command injection y TLS sobre HTTP

Atacante observado: 100.72.233.79.

Qué hizo: intentó parámetros tipo comando y envió handshakes TLS contra un puerto HTTP.

Respuesta: monitor automático detectó actividad anómala y aplicó bloqueo.

Lección: el monitoreo de patrones no estándar permite detectar pruebas manuales más allá de firmas conocidas.

C-05 · Rotación de IP y User-Agent falso

Atacante observado: 100.115.41.67.

Qué hizo: cambió de IP y usó User-Agent de iPad, generando requests a recursos internos.

Respuesta: el monitor detectó alta tasa y comportamiento anómalo.

Lección: la detección por comportamiento resiste mejor que la detección puramente por User-Agent.

Sección 14

Recomendaciones Priorizadas

Prioridad 1 · Crítico

  • Habilitar HTTPS y redirigir HTTP a HTTPS.
  • Corregir CORS con allowlist estricta y sin credenciales reflejadas.
  • Validar Host y headers proxy; no confiar en X-Forwarded-Proto de clientes directos.
  • Revisar REST API para endpoints con POST/PUT/PATCH/DELETE y exigir nonces/autenticación.

Prioridad 2 · Alta

  • Restringir /wp/v2/users, /batch/v1 y namespaces de plugins.
  • Proteger install.php, upgrade.php y repair.php.
  • Auditar Wordfence, Burst Statistics, AIOS, Advanced IP Blocker y WP-Optimize.
  • Corregir el error 500 de functions.php e investigar logs internos.

Prioridad 3 · Media

  • Configurar cookies con SameSite, Secure y HttpOnly.
  • Implementar CSP completa en todo el sitio, no solo en login.
  • Deshabilitar WP-Cron público y usar cron del sistema.
  • Corregir referencias a localhost:8080.
  • Crear robots.txt y security.txt.

Defensa propia

  • Aumentar bantime de fail2ban por encima de 90 segundos.
  • Corregir permisos de escritura de logs de mod_evasive.
  • Mantener honeypots y reglas ModSecurity actualizados.
  • Revisar dashboard periódicamente durante ventanas de ataque.
  • Documentar cada ban con IP, hora, patrón y regla activada.
Sección 15

Conclusión

El laboratorio permitió demostrar tanto capacidades defensivas como ofensivas. En la parte defensiva, el equipo construyó una arquitectura de protección por capas y corrigió fallos reales que impedían que algunas defensas funcionaran contra tráfico de Tailscale o contra intentos rápidos de login. El resultado fue un WordPress propio con controles de aplicación, red, monitoreo y engaño defensivo.

En la parte ofensiva, la auditoría contra 100.72.233.79:8080 encontró una postura de seguridad débil en varios aspectos fundamentales: CORS inseguro, ausencia de TLS, REST API expuesta, usuarios enumerables, plugins y namespaces públicos, endpoints administrativos accesibles y confianza indebida en headers manipulables por el cliente. El riesgo no proviene solo de cada hallazgo aislado, sino de su encadenamiento.

La principal enseñanza es que un WordPress aparentemente funcional puede quedar expuesto por configuraciones por defecto o inconsistentes. La defensa efectiva requiere validar cada control en el tráfico real, especialmente cuando hay Tailscale, proxies, headers de forwarding, REST API y plugins de seguridad interactuando.
