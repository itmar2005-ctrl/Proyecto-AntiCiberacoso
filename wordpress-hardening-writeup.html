04 — HALLAZGOS

Vulnerabilidades Críticas

CRIT-01
XML-RPC Habilitado
CRÍTICO
CVSS v3.1 Score9.8 / 10

El archivo xmlrpc.php de WordPress actúa como un endpoint de API remota que permite ejecutar acciones sobre el CMS mediante peticiones XML. En estado predeterminado, este endpoint estaba activo y accesible públicamente, lo que representa un vector de ataque de alto impacto.

  • Permite fuerza bruta sin límite de intentos (system.multicall)
  • Evasión de mecanismos anti-brute-force tradicionales
  • Ejecución remota de múltiples intentos en una sola petición HTTP
  • Abuso de métodos: wp.getUsersBlogs, wp.newPost
  • Pivoting para ataques DDoS amplificados (WordPress Pingback)
$ curl -X POST http://target/xmlrpc.php \
 -d '<?xml version="1.0"?>
 <methodCall><methodName>
 system.multicall
 </methodName>...</methodCall>'

HTTP/1.1 200 OK
▲ Endpoint activo. Sin protección.

CRIT-02
Enumeración de Usuarios Activos
CRÍTICO
CVSS v3.1 Score7.5 / 10

WordPress exponía el nombre de usuario del administrador a través de múltiples vectores de enumeración pasiva. El nombre de usuario obtenido reduce el espacio del ataque de fuerza bruta a una sola dimensión (solo la contraseña).

USUARIO DESCUBIERTO

Encontrado vía: REST API /wp-json/wp/v2/users, Author archive, WPScan passive detection
  • API REST /wp-json/wp/v2/users devuelve JSON con IDs y usernames
  • Parámetro ?author=1 redirige a URL con slug del usuario
  • Meta tags del tema revelan el nombre del autor

MED-03
wp-login.php Expuesto Sin Protección
MEDIO
CVSS v3.1 Score5.3 / 10

El formulario de autenticación de WordPress era accesible sin ningún tipo de limitación de intentos, CAPTCHA, 2FA o restricción por IP. Combinado con el usuario administrador conocido (CRIT-02), esto constituyó la base del ataque de fuerza bruta documentado.

  • Sin rate-limiting en intentos de login
  • Sin CAPTCHA ni mecanismo anti-bot
  • Sin doble factor de autenticación
  • Sin restricción geográfica o por IP

11 — CONTROLES

Hardening de WordPress

CONTROL 1 — Bloqueo de XML-RPC (.htaccess / Apache VHost)
<Files xmlrpc.php>
  Require all denied
</Files>

Resultado: cualquier solicitud a xmlrpc.php retorna HTTP 403 independientemente del método o payload.

CONTROL 2 — Protección de wp-config.php
<Files wp-config.php>
  Require all denied
</Files>

Previene el acceso directo al archivo de configuración que contiene credenciales de base de datos, claves secretas y configuración del CMS.

CONTROL 3 — Bloqueo de Enumeración de Usuarios vía ?author=
RewriteEngine On
RewriteCond %{QUERY_STRING} (^|&)author=[0-9]+(&|$) [NC]
RewriteRule ^ - [F,L]

# Bloqueo adicional vía REST API
# En functions.php:
add_filter('rest_endpoints', function($endpoints) {
  if (isset($endpoints['/wp/v2/users'])) {
    unset($endpoints['/wp/v2/users']);
  }
  return $endpoints;
});

14 — CIERRE TÉCNICO

Conclusiones Técnicas

Lecciones Aprendidas
  • La superficie de ataque de WordPress predeterminado es amplia y bien documentada por los atacantes.
  • Los ataques de fuerza bruta son automáticos, inmediatos y masivos; sin Fail2Ban, son imparables.
  • La enumeración de usuarios es un paso previo crítico que multiplica la efectividad del brute-force.
  • ModSecurity detiene amenazas que no son visibles en logs estándar de Apache.
  • El monitoreo en tiempo real cambia la postura de reactiva a proactiva.
  • Las actualizaciones no son opcionales en un entorno expuesto a Internet.
Recomendaciones Finales
  • Deshabilitar XML-RPC si no es requerido por plugins.
  • Implementar autenticación en dos pasos (2FA) para wp-admin.
  • Utilizar contraseñas generadas con gestores de contraseñas.
  • Mantener WordPress, plugins y temas siempre actualizados.
  • Implementar HTTPS obligatorio con HSTS.
  • Aplicar política de mínimo privilegio en usuarios de BD.
  • Automatizar escaneos periódicos con WPScan en modo defensa.
  • Considerar un WAF en capa de CDN (Cloudflare, AWS WAF).
CONCLUSIÓN GENERAL

Este laboratorio demostró que una instalación predeterminada de WordPress presenta una superficie de ataque significativa, explotable con herramientas de libre acceso en minutos. Sin embargo, la implementación sistemática de controles de seguridad —desde reglas Apache básicas hasta un WAF completo y un sistema de detección de intrusiones— reduce el riesgo a niveles aceptables de forma documentable y verificable.

El ciclo completo de reconocimiento → explotación → defensa → validación refleja la metodología real utilizada en entornos de operaciones de seguridad (SOC/NOC), y evidencia que la defensa efectiva no es un estado estático, sino un proceso continuo de monitoreo, respuesta y mejora.
